# HMCTS Task Management API (Backend)

A REST API that lets caseworkers keep track of their tasks. Built for the HMCTS DTS
Developer Technical Test.

It supports creating a task, retrieving a single task or all tasks, updating a task's
status, and deleting a task. Data is persisted in a database, inputs are validated, and
errors are returned in a consistent structure.

## How to run

> **Note:** The `cd` paths below are examples. Change them to wherever you cloned
> the two repositories on your machine.

This solution has two parts that run together: the **backend** (this API) and the
**frontend** (web UI). Run each in its own terminal, and start the backend first.

### 1. Backend (start first) — http://localhost:4000

```bash
cd /Users/user/HMCTSTest/hmcts-dev-test-backend
./gradlew bootRun
```

### 2. Frontend — http://localhost:3100

```bash
cd /Users/user/HMCTSTest/hmcts-dev-test-frontend
yarn install   # first time only
yarn build     # first time, or after changes
yarn start
```

Then open **http://localhost:3100** in your browser (it redirects to the task list).

- API docs (Swagger UI): http://localhost:4000/swagger-ui.html
- Start the backend before the frontend, because the frontend calls the backend API.
- If a port is already in use, free it with `lsof -ti tcp:4000 | xargs kill -9`
  (or `tcp:3100` for the frontend).

## Tech stack

- Java 21
- Spring Boot 4 (Spring Web, Spring Data JPA, Bean Validation, Actuator)
- H2 in-memory database
- springdoc-openapi (Swagger UI)
- Gradle
- JUnit 5, Mockito, RestAssured

## Prerequisites

- JDK 21 (the Gradle toolchain will select it)

No local database or Docker is required — H2 runs in-memory.

## Running the application

```bash
./gradlew bootRun
```

The service starts on **http://localhost:4000**.

To build a runnable jar:

```bash
./gradlew build
java -jar build/libs/test-backend.jar
```

## Database

An in-memory H2 database is created on startup and reset when the application stops.
The H2 web console is available at http://localhost:4000/h2-console using:

| Setting  | Value                                              |
|----------|----------------------------------------------------|
| JDBC URL | `jdbc:h2:mem:tasks;DB_CLOSE_DELAY=-1;DB_CLOSE_ON_EXIT=FALSE` |
| User     | `sa`                                               |
| Password | *(empty)*                                          |

## Endpoints

Base path: `/api/tasks`

| Method   | Path                       | Description                | Success |
|----------|----------------------------|----------------------------|---------|
| `POST`   | `/api/tasks`               | Create a task              | 201     |
| `GET`    | `/api/tasks`               | Retrieve all tasks         | 200     |
| `GET`    | `/api/tasks/{id}`          | Retrieve a task by id      | 200     |
| `PATCH`  | `/api/tasks/{id}/status`   | Update the status of a task| 200     |
| `DELETE` | `/api/tasks/{id}`          | Delete a task              | 204     |

### Task model

| Field         | Type            | Required | Notes                                   |
|---------------|-----------------|----------|-----------------------------------------|
| `id`          | number          | —        | Generated by the server                 |
| `title`       | string          | yes      | Max 255 characters                      |
| `description` | string          | no       | Max 2000 characters                     |
| `status`      | string (enum)   | yes      | One of the status values below          |
| `dueDateTime` | ISO date-time   | yes      | e.g. `2026-07-10T09:00:00`              |

### Status values

`PENDING`, `IN_PROGRESS`, `COMPLETED`, `CANCELLED`

### Example requests

Create a task:

```bash
curl -X POST http://localhost:4000/api/tasks \
  -H "Content-Type: application/json" \
  -d '{
        "title": "Review case file",
        "description": "Check all documents",
        "status": "PENDING",
        "dueDateTime": "2026-07-10T09:00:00"
      }'
```

Update a task's status:

```bash
curl -X PATCH http://localhost:4000/api/tasks/1/status \
  -H "Content-Type: application/json" \
  -d '{ "status": "COMPLETED" }'
```

Delete a task:

```bash
curl -X DELETE http://localhost:4000/api/tasks/1
```

### Error responses

Errors are returned in a consistent shape:

```json
{
  "status": 400,
  "message": "Validation failed",
  "fieldErrors": { "title": "title is required" },
  "timestamp": "2026-07-10T09:00:00"
}
```

- `400 Bad Request` — validation failure or malformed/invalid body
- `404 Not Found` — no task exists for the given id

## Testing

```bash
./gradlew test         # unit and web-layer tests
./gradlew integration  # full-stack integration tests (real HTTP + H2)
./gradlew check        # runs the full verification suite
```

Test coverage focuses on functional behaviour plus edge and negative cases:

- `TaskServiceTest` — service logic with a mocked repository (JUnit 5 + Mockito)
- `TaskRepositoryTest` — JPA persistence against H2 (`@DataJpaTest`)
- `CreateTaskRequestValidationTest` — validation constraints
- `TaskControllerTest` — REST layer with a mocked service (`@WebMvcTest`), incl. 400/404
- `TaskIntegrationTest` / `ApiDocsIntegrationTest` — full stack and API docs
